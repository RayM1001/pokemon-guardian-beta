# 測驗規則設計說明

> 文件：`reports/rules_design_notes.md`  
> 版本：1.1.0 | 日期：2026-05-23

---

## 1. 為什麼把規則獨立成 rules.json

### 資料與邏輯分離

`data/pokemon.json` 是純粹的「資料層」，記錄每隻寶可夢的既有屬性（名稱、五行、特性、人氣等）；`data/rules.json` 是「規則層」，定義測驗問題、評分邏輯、場景條件與呈現語氣。

兩者分離的好處：

- **可獨立修改**：調整題目措辭、改變分數權重，不需要重新跑 1025 筆的轉換腳本。
- **語系可擴充**：想新增泰文或韓文版本，只需在 `rules.json` 的 locale 與各題目區塊新增對應欄位，不影響寶可夢資料。
- **推薦邏輯可版本化**：`rules.json` 帶有 `_meta.version`，可以同時維護多個版本的評分邏輯，供 A/B 測試使用。
- **前端與後端共用**：`rules.json` 可作為前端靜態規則讀取，也可搬到後端 API 返回，不綁定任何框架。

## 2. 五行、心理測驗、場景、人氣分數如何一起影響推薦

整個評分系統由四個維度疊加，最終加總排序：

```
總分 = (出生五行得分 × 0.35)
     + (心理測驗標籤得分 × 0.25)
     + (場景適合得分 × 0.25)
     + (人氣加分 × 0.15)
     - 扣分項
```

### 各維度的運作方式

**出生五行（35%）**
使用者輸入出生月份、生日個位數、時辰，系統累積各五行的傾向分數，判斷出「最弱的五行」作為需要補足的方向。寶可夢的 `fiveElement.main` 若符合該補足方向，給予最高加分（+10）。這個維度佔比最大，因為它反映了相對穩定的個人底色。

**心理測驗標籤（25%）**
五道測驗題目各自帶有 tags（如「修復」「行動」「守護」等），使用者的答案累積出一組「當前心理狀態標籤池」。系統比對寶可夢的 `keywords` 與 `psychStates` 欄位，每命中一個標籤 +3 分。這個維度反映的是「今天的狀態」，比出生五行更動態、更短期。

**場景適合度（25%）**
由測驗第三題（放置位置）決定場景，每個場景都有偏好的五行、陰陽與 boostTags/avoidTags。五行符合 +6，陰陽符合 +4，boostTags 命中每個 +3，avoidTags 命中每個 -5。這個維度確保推薦出來的寶可夢「放在那個位置不違和」。

**人氣加分（15%）**
直接使用 pokemon.json 的 `popularityScore`（1–5 分），轉成 +1 到 +5。這個維度最低，原因見下一節。

**扣分機制**
- 使用者已有強烈的火偏向，還推薦陽火屬性寶可夢：-5（避免過度堆疊同一五行）
- 使用者水偏強，推薦陰水寶可夢放在非臥室場景：-3（水偏強的臥室才是加分情境）
- 場景 avoidTags 命中：每個 -5（硬性約束，排除明顯不適合的場景搭配）

## 3. 為什麼人氣分數不能權重過高

### 人氣高≠適合這個人

人氣評分來自公開問卷與對戰使用率資料，反映的是「大多數玩家喜愛的寶可夢」，而非「適合這位使用者當前狀態的寶可夢」。如果人氣權重過高，推薦結果會集中在皮卡丘、噴火龍等少數熱門寶可夢，失去個人化的意義。

### 長尾效應的保留

1025 隻寶可夢中，大量中低人氣的寶可夢（評分 2–3）在五行、心理狀態、場景上可能是非常精準的配對。如果人氣佔比過高，這些「冷門但精準」的寶可夢永遠排不上來，整個推薦系統等同縮水成「熱門寶可夢排行榜」。

### 目前設定（15%）的角色

在同分或接近滿分的情況下，人氣作為最後的「平局決勝」依據，略為偏向使用者較熟悉的寶可夢，降低結果頁出現完全陌生名字的機率。這是輔助角色，而非主導角色。

## 4. 為什麼保留本命夥伴，不否定使用者原本喜歡的寶可夢

### 尊重情感連結

寶可夢的核心精神是「夥伴」。使用者對特定寶可夢的喜愛往往來自童年記憶、遊戲經歷或純粹的角色魅力，這些情感連結是真實且有意義的，無法被算法評分「否定」。

### 推薦的角色是「補充」而非「替換」

測驗的定位是幫助使用者找到「在這個時期、這個空間，可以陪伴你的守護夥伴」，而不是宣稱「你應該喜歡這隻，不應該喜歡那隻」。`favoritePartner` 槽位與 `mainGuardian` 並列展示，讓使用者可以看到「你一直喜歡的」和「系統現在推薦的」的差異，這種對照本身就是有趣的體驗。

### 避免產生抗拒感

如果推薦系統「否定」使用者的本命，使用者最直接的反應是不信任整個測驗。保留本命夥伴的設計讓使用者感到被尊重，增加對推薦結果的開放度。

## 5. 為什麼官方 30 週年圖片只做外部連結，不直接嵌入

### 版權合規

Pokémon 30 週年圖示屬於 The Pokémon Company 的官方素材，未獲授權不得用於第三方應用的嵌入式展示或重新散布。直接將 `official30th.link` 作為 `<img src>` 使用，等同於在自己的網頁上托管並展示官方圖片，可能構成著作權侵害。

### 技術層面的隔離

`pokemon.json` 中每筆資料的 `official30th.previewAllowed` 固定為 `false`，作為程式碼層級的明確標記，提醒所有前端開發者「這個欄位不用來顯示圖片」。`rules.json` 的 `officialLinkPolicy.frontendRule` 也提供明確的實作約束說明。

### 提供外部連結的好處

將使用者導向官方 30 週年頁面，反而是一種更好的使用者體驗設計：
- 使用者可以在官方情境下欣賞高品質圖片
- 官方頁面可能有動畫或互動效果，比靜態嵌入更豐富
- 支持了 Pokémon 30 週年的官方宣傳流量

### 非官方企劃的自我保護

保持合規的外部連結設計，讓這個非官方粉絲企劃可以長期運作，不因版權問題被下架，最終對所有使用者都更有利。

## 6. Quiz Tags 的設計：為何採用繁中內部標籤

### 標籤是計分引擎的內部語言，不是顯示語言

`quizQuestions` 中每個選項都帶有 `tags` 陣列，例如：

```json
"tags": ["安定", "修復", "低刺激"]
```

這些 tags **全部採用繁體中文字串**，無論使用者選擇哪個語系（zh_tw / en / ja），tags 的值都保持不變。

### 為什麼不隨語系切換 tags？

評分邏輯的核心是：將使用者測驗答案的 tags 與 pokemon.json 中每隻寶可夢的 `keywords` 和 `psychStates` 欄位做字串比對。

`pokemon.json` 的這兩個欄位只有繁中版本（資料來源即為繁中），例如：

```json
"keywords":    ["成長", "修復", "安定", "效率", "守護"],
"psychStates": ["療癒", "成長", "茁壯", "警覺", "防衛"]
```

若 tags 也隨語系切換成英文或日文，就無法與 pokemon.json 做精確比對，計分系統會完全失效。

### 語系只影響顯示層，不影響計分層

正確的分層如下：

| 層次 | 語系相關 | 說明 |
|------|----------|------|
| 題目文字（`question`、`text`） | ✅ 三語系各自翻譯 | 使用者看到的問題與選項 |
| 選項 tags | ❌ 固定繁中 | 計分引擎的內部標籤，不顯示給使用者 |
| pokemon.json keywords/psychStates | ❌ 固定繁中 | 資料層，不因語系改變 |
| 結果頁說明文字 | ✅ 依 locale 選擇 | 從 fiveElements / sceneRules 的三語系欄位取出 |

### 出生時辰的選填設計（v1.1 補充）

`birthElementRules.hourRules.birthTime` 明確標記 `optional: true`，對應三語系的「不知道／略過」按鈕文案。

前端實作規則：
- 當使用者選擇略過，傳入 `unknownValue: "unknown"` 給計分邏輯
- 計分邏輯接收到 `"unknown"` 時，該維度得分設為 `unknownScore: 0`，不加分也不扣分
- **不得阻擋使用者繼續進行測驗**，出生時辰略過不影響其他三個維度的正常計分

## 7. 五行相生相剋關係（v1.2 新增）

### 設計目的

直接補弱五行雖然直觀，但只做直接補缺會讓推薦結果太過單一。加入相生相剋關係後，引擎可以找到「間接幫助」使用者的寶可夢，增加推薦的層次感。

### 相生：間接補位加分

五行相生循環：**水 → 木 → 火 → 土 → 金 → 水**

- 若使用者 `primaryNeed = 土`，則「火生土」→ 火行寶可夢可以間接滋養土，給予 **+4 間接補位分**
- 若使用者 `secondaryNeed = 木`，則「水生木」→ 水行寶可夢可間接補位，給予 **+2 間接補位分**
- 相生加分**不可高於直接補缺**（+10），也不可高於場景五行加分（+6）

### 相剋：溫和調節過旺狀態（+2）

五行相剋循環：**木剋土 → 土剋水 → 水剋火 → 火剋金 → 金剋木**

- 若使用者某五行分數明顯偏旺（如火最強），對應可以調節火的五行（水）給予 **+2 平衡加分**
- 這是最低優先的計分項，只作為微調，**不主導推薦**
- 相剋調節可能與場景規則衝突（例如火旺但場景是臥室時，水可能因臥室-陰水規則被調整），以場景規則優先

### 文案用語規範

| 禁止用語 | 替換用語 |
|---------|---------|
| 剋你 | 平衡、降溫 |
| 壓制 | 整理過旺狀態 |
| 不適合 | 今日由其他寶可夢協助 |
| 火系能量 | 五行上的暖火感 |
| 水系氣息 | 五行上的柔水感 |

## 8. tagElementMap：quiz tags 到五行的橋梁（v1.1 新增）

`tagElementMap` 是推薦引擎在 `quizTags`（使用者測驗答案標籤）和五行計分之間的對照表。

### 主要用途

1. **watch 場景動態五行推導**：當 `sceneRules.watch.dynamicElementMode = true` 時，不使用固定 `suitableElements`，改以 `quizTags` 透過 `tagElementMap` 累積計算出當下偏向的五行，作為 watch 場景的 `suitableElements` 替代品。
2. **輔助 deriveElementNeeds**：未來可擴充支援從 quiz 答案反推補充五行需求。

### 設計原則

- 所有 tag 鍵值均為繁中內部標籤，不隨語系切換（見第 6 節說明）
- 同一個 tag 只歸屬一個五行，避免多重命中造成計分混亂
- 五行覆蓋盡量均衡，每行 5–10 個 tag

| 五行 | 代表標籤（部分） |
|------|-----------------|
| 木 | 成長、學習、啟動、舒展、新開始 |
| 火 | 行動、推進、熱情、爆發力 |
| 土 | 安定、穩定、守護、安心、陪伴、人際 |
| 金 | 專注、效率、秩序、決策、清晰 |
| 水 | 修復、睡眠、柔和、流動、低刺激、直覺 |

## 附錄：規則版本調整建議

以下是未來可以根據用戶反饋調整的規則參數，不需改動 pokemon.json：

| 項目 | 目前值 | 調整方向 |
|------|--------|----------|
| `birthElementWeight` | 0.35 | 若用戶覺得出生資料影響太大，可降至 0.25–0.30 |
| `quizStateWeight` | 0.25 | 若更強調當下感受，可提升至 0.35 |
| `keywordTagHit` 加分 | +3/個 | 若結果過於分散，可提升至 +5 |
| `sceneAvoidTagHit` 扣分 | -5/個 | 若過濾太激烈，可放寬至 -3 |
| 題目數量 | 5 題 | 可擴充至 7–8 題以增加精準度 |

*本設計說明由 `data/rules.json` 對應，如修改規則請同步更新本文件。*
